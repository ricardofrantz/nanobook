# v0.15 Kill-Gate Criteria

**Status**: PRE-COMMITTED (written before v0.13 work begins)
**Date**: 2026-05-12
**Updated**: 2026-05-13 (renumbered from v0.14 to v0.15 after OCaml oracle released as v0.14)
**Purpose**: Prevent confirmation bias by committing to kill-gate thresholds before observing data

## Context

v0.13 (Ops Hardening) completed failure-injection testing for 9 failure modes. v0.14 (OCaml Oracle) added dual-language differential testing. v0.15 (Paper Trading Soak) is a 4-6 week operational test that validates the system against real market conditions.

This document commits to explicit kill-gate criteria for v0.15 **before** any v0.13 work begins. The criteria are unfalsifiable if written after observing the data.

## Kill-Gate Decision

The decision to proceed from v0.14 to v0.15 will be evaluated against four criteria. If **any** criterion fails, v0.15 will be cancelled and the project will be re-evaluated.

### Criterion 1: Sustained Engagement

**Metric**: Hours/week on nanobook over the prior 8 weeks
**Threshold**: Kill if `< 5 hours/week`
**Measurement**: Calendar weeks, any nanobook-related work (coding, docs, research, ops)
**Rationale**: <5 hours/week indicates loss of momentum or competing priorities. A 4-6 week operational soak requires sustained engagement.

### Criterion 2: Operational Debt

**Metric**: Unplanned v0.13 ops fixes
**Threshold**: Kill if `≥ 3` unplanned ops fixes
**Definition**: "Unplanned ops fix" = a bug or issue discovered during normal operation (not during testing) that requires a patch or hotfix. Excludes:
- Bugs found during development testing
- Documentation updates
- Performance tuning (non-critical)
**Rationale**: ≥3 unplanned ops fixes indicates systemic instability that a paper soak will expose catastrophically.

### Criterion 3: Technical Blockers

**Metric**: Working days lost to unexpected technical blockers in prior release
**Threshold**: Kill if `> 5 working days` lost
**Definition**: "Working day" = 8 hours of productive time. "Technical blocker" = any environment, toolchain, or dependency issue that blocks progress on the planned work (e.g., opam switch failures, dune build errors, missing dependencies, CI infrastructure issues).
**Rationale**: If technical setup costs exceed 5 days, the ROI is negative for proceeding with a 4-6 week operational soak.
**Note**: v0.14 OCaml oracle setup completed without major blockers (<1 day lost to library naming issue).

### Criterion 4: Burnout Self-Report

**Metric**: Subjective burnout assessment
**Threshold**: Hard stop on self-report
**Definition**: If the maintainer self-reports burnout or loss of motivation, v0.14 will be cancelled regardless of other criteria.
**Rationale**: Open-source maintenance is voluntary. No project is worth burnout.

## Evaluation Process

1. **Timing**: Evaluation occurs after v0.14 completion, before v0.15 kickoff
2. **Data Collection**:
   - Criterion 1: Git commit timestamps + calendar tracking
   - Criterion 2: Git log filtered for "ops fix" commits + issue tracker
   - Criterion 3: Time tracking during v0.14 (OCaml oracle setup and implementation)
   - Criterion 4: Self-assessment
3. **Decision Rule**: If any criterion fails, v0.15 is cancelled
4. **Transparency**: Results will be documented in a follow-up doc (`docs/solutions/v0.15-kill-gate-evaluation.md`)

## Anti-Patterns

These criteria are designed to prevent common failure modes:

- **Sunk cost fallacy**: "We've come this far, we should continue" → countered by objective thresholds
- **Optimism bias**: "It'll probably work out" → countered by pre-commitment
- **Scope creep**: "Let's just add one more feature" → countered by burnout check
- **Technical debt denial**: "It's just a few bugs" → countered by ops fix count

## Revision Policy

This document may be revised **only** before v0.13 work begins. Once v0.13 starts, the criteria are locked. Any revision must:
1. Be committed to git with rationale
2. Maintain or strengthen the thresholds (not weaken them)
3. Be documented in this file's revision history
